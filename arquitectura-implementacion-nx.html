Arquitectura de Implementación NX®

Versión: 1.0

Uso: Desarrollo, despliegue, seguridad y operación técnica

Este documento define la arquitectura técnica recomendada para implementar la plataforma Netinho Xperience como producto digital escalable, mobile-first y orientado a decisiones patrimoniales.

1. Stack tecnológico recomendado

Capa Herramienta recomendada Propósito
Frontend Next.js + Tailwind CSS Interfaz rápida, responsive, SEO/SXO y experiencia mobile-first.
Hosting Vercel Despliegue gratuito inicial, SSL, rendimiento y escalabilidad.
Base de datos Supabase PostgreSQL Usuarios, expedientes, pagos, informes, logs e indicadores.
Storage Supabase Storage Almacenamiento temporal de documentos, fotografías y PDFs.
Autenticación Supabase Auth Acceso seguro de usuarios y recuperación de cuenta.
IA Gemini API Lectura, análisis, clasificación y apoyo en generación del informe.
Pagos Niubiz o pasarela equivalente Validación automática de pagos por servicio.
PDF Generador de PDF server-side Emisión del Informe de Decisión Patrimonial NX®.
Validación Página pública con código único Verificar autenticidad, vigencia y estado del informe.

5. Flujo técnico principal

Fase Acción técnica
1. Usuario inicia evaluación Frontend captura tipo de activo y muestra modal previo al pago.
2. Usuario acepta condiciones Se registra consentimiento previo.
3. Pago Niubiz procesa el pago y retorna estado.
4. Creación de expediente Supabase crea código único NX y estado inicial.
5. Carga de evidencias Storage recibe archivos y los asocia al expediente.
6. Validación de calidad Se verifica tipo, tamaño, legibilidad y completitud mínima.
7. Procesamiento IA Gemini analiza información y genera hallazgos estructurados.
8. Cálculo de indicadores Se calculan IR-NX, II-NX, IPD-NX, CE-NX y NCI-NX.
9. Emisión de informe Se genera PDF con código único, marca de agua, vigencia y descargos.
10. Validación pública La página validar permite verificar código, estado y vigencia.

6. Estados del expediente

Estado Significado
pago_pendiente El usuario inició el proceso, pero el pago aún no fue validado.
activo Pago validado y expediente habilitado.
carga_pendiente Faltan evidencias o documentos.
carga_completa El expediente alcanzó el mínimo necesario.
en_analisis El expediente está siendo procesado.
informe_emitido El informe fue generado.
vencido El plazo de carga o vigencia del informe expiró.
rechazado El expediente fue rechazado por causa técnica, legal o de integridad.

7. Seguridad mínima obligatoria

  • HTTPS obligatorio.
  • Claves privadas fuera del frontend.
  • Control de acceso por usuario.
  • URLs firmadas para documentos sensibles.
  • Eliminación programada de evidencias.
  • Logs de actividad.
  • Validación de tamaño y tipo de archivo.
  • Protección contra carga maliciosa.
  • Política de privacidad visible.

8. Reglas de almacenamiento

Elemento Tiempo recomendado

9. Integración con documentos legales

La plataforma debe enlazar siempre a:

  • terminos.html

  • consentimiento.html
  • cookies.html
  • reembolsos.html
  • reclamaciones.html

10. Principio de implementación MVP

El MVP debe validar el flujo completo con la menor complejidad posible: pago, expediente, carga, informe y validación. Toda funcionalidad adicional debe incorporarse después de validar uso real y disposición de pago.

11. Riesgos técnicos principales

Riesgo Mitigación
Falla de pasarela de pago Logs, validación manual excepcional y política de reembolso.
Archivos ilegibles Validación previa y solicitud de recarga.
IA genera lectura insuficiente Indicadores de incertidumbre y descargos claros.
Exposición de datos Seguridad, almacenamiento limitado y accesos restringidos.
Escalamiento rápido Arquitectura desacoplada y monitoreo.
