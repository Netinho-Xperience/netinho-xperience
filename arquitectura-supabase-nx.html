Arquitectura Supabase NX®

Versión: 1.0

Uso: Base de datos, autenticación, almacenamiento, seguridad y operación

Este documento define la arquitectura recomendada de Supabase para soportar la plataforma Netinho Xperience: expedientes, usuarios, pagos, evidencias, indicadores, informes, validaciones y trazabilidad.

1. Objetivo

Supabase será utilizado como capa de datos, autenticación, almacenamiento de evidencias y soporte operativo de la plataforma NX Decision Engine®.

2. Módulos Supabase utilizados

Módulo Uso en NX
PostgreSQL Base de datos relacional para usuarios, expedientes, pagos, indicadores, informes y logs.
Auth Registro, inicio de sesión, recuperación de cuenta y asociación de expedientes al usuario.
Storage Almacenamiento temporal de documentos, fotografías, videos y PDFs cargados.
Row Level Security Control de acceso para que cada usuario solo vea sus propios expedientes.
Edge Functions Procesos server-side: creación de expediente, validación de pagos, procesamiento de informes y limpieza programada.

3. Principios de datos

  • Minimización: pedir solo los datos necesarios.
  • Finalidad: usar datos únicamente para el servicio contratado.
  • Trazabilidad: registrar eventos relevantes del expediente.
  • Seguridad: no exponer claves privadas en frontend.
  • Temporalidad: eliminar evidencias conforme a la política de privacidad.
  • Escalabilidad: separar usuarios, expedientes, pagos, evidencias e informes.

4. Tablas principales

Tabla Propósito
profiles Datos básicos del usuario asociado a Supabase Auth.
expedientes Registro central de cada evaluación NX.
pagos Control de pagos realizados mediante Niubiz u otra pasarela.
evidencias Archivos cargados por el usuario: fotos, PDFs, videos y documentos.
indicadores Resultados numéricos: riesgo, incertidumbre, preparación, calidad y confianza.
informes PDF emitido, código de validación, vigencia y estado.
logs_expediente Bitácora de eventos del expediente.
reclamos_soporte Consultas o incidencias vinculadas a expedientes.

5. Estados del expediente

Estado Descripción
pago_pendiente El usuario inició el flujo, pero el pago aún no fue validado.
activo Pago validado y expediente creado.
carga_pendiente Faltan evidencias o documentos requeridos.
carga_completa El usuario cargó la información mínima necesaria.
en_analisis La información está siendo procesada.
informe_emitido Informe generado y disponible para descarga.
vencido Expiró el plazo de carga o la vigencia del informe.
rechazado Expediente rechazado por riesgo, integridad, fraude o imposibilidad técnica.

6. Diseño SQL base sugerido

create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  nombres text,
  apellidos text,
  tipo_documento text,
  numero_documento text,
  correo text,
  created_at timestamptz default now()
);

create table expedientes (
  id uuid primary key default gen_random_uuid(),
  codigo text unique not null,
  user_id uuid references auth.users(id),
  tipo_activo text not null,
  estado text default 'pago_pendiente',
  fecha_pago timestamptz,
  fecha_activacion timestamptz,
  fecha_vencimiento_carga timestamptz,
  fecha_emision timestamptz,
  fecha_vencimiento_informe timestamptz,
  created_at timestamptz default now()
);

create table pagos (
  id uuid primary key default gen_random_uuid(),
  expediente_id uuid references expedientes(id) on delete cascade,
  monto numeric(10,2) not null,
  moneda text default 'PEN',
  pasarela text default 'Niubiz',
  estado_pago text default 'pendiente',
  codigo_operacion text,
  fecha_pago timestamptz,
  created_at timestamptz default now()
);

create table evidencias (
  id uuid primary key default gen_random_uuid(),
  expediente_id uuid references expedientes(id) on delete cascade,
  tipo_evidencia text not null,
  tipo_archivo text,
  nombre_archivo text,
  storage_path text,
  estado_revision text default 'recibido',
  observacion text,
  created_at timestamptz default now()
);

create table indicadores (
  id uuid primary key default gen_random_uuid(),
  expediente_id uuid references expedientes(id) on delete cascade,
  riesgo_nx integer,
  incertidumbre_nx integer,
  preparacion_nx integer,
  calidad_expediente integer,
  confianza_informe integer,
  created_at timestamptz default now()
);

create table informes (
  id uuid primary key default gen_random_uuid(),
  expediente_id uuid references expedientes(id) on delete cascade,
  codigo_validacion text unique not null,
  version text default '1.0',
  url_pdf text,
  estado text default 'vigente',
  fecha_emision timestamptz default now(),
  fecha_vencimiento timestamptz,
  created_at timestamptz default now()
);

create table logs_expediente (
  id uuid primary key default gen_random_uuid(),
  expediente_id uuid references expedientes(id) on delete cascade,
  evento text not null,
  detalle text,
  created_at timestamptz default now()
);

7. Buckets de Storage

Bucket Uso Acceso recomendado
evidencias Fotos, videos y documentos cargados por el usuario. Privado.
informes PDF final emitido por NX. Privado con URL firmada.
assets-publicos Logos, ejemplos de fotos y recursos públicos. Público.

8. Reglas de seguridad

  • Activar Row Level Security en tablas sensibles.
  • El usuario solo puede ver sus propios expedientes.
  • Los archivos sensibles deben almacenarse en buckets privados.
  • Las URLs de descarga deben ser temporales.
  • Las claves service_role solo deben usarse en servidor.
  • No mostrar documentos sensibles en páginas públicas.

9. RLS conceptual

-- Ejemplo conceptual:
-- El usuario solo puede leer expedientes donde user_id = auth.uid()

create policy "Usuarios ven sus expedientes"
on expedientes
for select
using (auth.uid() = user_id);

10. Logs obligatorios

Cada expediente debe registrar como mínimo:

  • pago iniciado;
  • pago validado;
  • expediente creado;
  • archivo cargado;
  • carga completa;
  • análisis iniciado;
  • informe emitido;
  • informe descargado;
  • expediente vencido;
  • incidencia o reclamo.

11. Conservación y eliminación

Elemento Plazo recomendado Acción posterior
Evidencias 90 días Eliminar o anonimizar.
Informes 24 meses Archivar o eliminar según política vigente.
Códigos de validación 24 meses Marcar como vencido o eliminar.
Logs 24 meses Archivar o depurar.

12. Funciones Edge recomendadas

Función Propósito
crear-expediente Crear expediente luego del pago validado.
validar-pago-niubiz Confirmar estado del pago y actualizar expediente.
procesar-evidencias Enviar archivos al motor IA y registrar hallazgos.
generar-informe Construir PDF final y guardarlo.
validar-informe Responder estado público del código único.
limpiar-evidencias Eliminar evidencias vencidas según política de privacidad.

13. Validación pública

La página de validación pública no debe exponer datos sensibles. Solo debe mostrar:

  • estado del informe;
  • código;
  • fecha de emisión;
  • fecha de vencimiento;
  • tipo de activo;
  • solicitante parcialmente anonimizado;
  • versión del informe.

14. Recomendación de implementación

Construir primero el flujo mínimo: usuario, pago, expediente, carga de evidencias, informe manual-asistido, validación pública. Luego automatizar progresivamente el análisis, indicadores y generación avanzada del PDF.
