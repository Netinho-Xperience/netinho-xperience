5. Despliegue

GitHub → Vercel → Supabase → Producción

Cada push a main deberá desplegar producción. Cada push a develop deberá desplegar un ambiente de prueba.

6. Variables de entorno

VariableUso
NEXT_PUBLIC_SUPABASE_URLURL pública de Supabase.
NEXT_PUBLIC_SUPABASE_ANON_KEYClave pública Supabase.
SUPABASE_SERVICE_ROLE_KEYClave privada del servidor. Nunca exponer en frontend.
GEMINI_API_KEYClave de Gemini API.
NIUBIZ_MERCHANT_IDID del comercio.
NIUBIZ_ACCESS_KEYCredencial Niubiz.
NIUBIZ_SECRET_KEYClave privada Niubiz.
NEXT_PUBLIC_APP_URLURL pública de la aplicación.

7. Reglas de seguridad

  • No subir archivos .env a GitHub.
  • No exponer claves privadas en componentes frontend.
  • Usar variables de entorno en Vercel.
  • Separar ambiente de pruebas y producción.
  • Activar Row Level Security en Supabase.
  • Usar buckets privados para evidencias e informes.
  • Registrar logs de eventos críticos.

10. Checklist antes de producción

  • Landing responsive.
  • Pago probado.
  • Expediente creado correctamente.
  • Temporizador funcional.
  • Carga de evidencias funcional.
  • Indicadores calculados.
  • PDF generado con marca de agua.
  • Código único validable.
  • Libro de Reclamaciones operativo.
  • Términos, privacidad, descargos y reembolsos visibles.

11. Regla final de implementación

No se debe pasar a producción ningún módulo que no tenga estado, logs, manejo de error, texto claro para el usuario y enlace a soporte.
